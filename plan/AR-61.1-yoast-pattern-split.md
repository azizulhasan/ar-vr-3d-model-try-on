# AR-61 §1.1 — Yoast-pattern Free / Pro split

The wp.org closure flagged compression as Guideline 5 trialware. Reading the closure email's
"more issues of the same nature" warning literally, **every Pro-gate in Free is at risk** —
not just the compression one. This plan applies the
[free-pro-architecture-pattern](../../../text-to-audio/plan/tickets/free-pro-architecture-pattern.md)
(the Yoast-derived blueprint we used on `text-to-audio`) to the AR plugin pair.

> **The rule:** the free plugin must be 100% functional on its own. Pro **adds** code that is
> not in the free ZIP — it never **unlocks** code that is.

Goal: ship a Free that has no `WP_Error('pro_only')`, no count caps, no silent feature
downgrades, and no "🔒 Upgrade to Pro" disabled controls. Pro continues to add advanced
features through filters, never by being granted access to disabled Free code.

## Backward-compatibility rules (MUST follow in every phase)

A live AtlasAR install can be one of four states the day a new version
ships. Every change in this plan must keep all four working without
fatal errors and without sudden feature loss.

| # | Scenario | Risk |
|---|---|---|
| 1 | **Free user, Free updates** | Old free models / settings must keep working. No "your data is gone" experience. |
| 2 | **Pro user, Free updates first** (Free auto-update, Pro still old) | Pro consumes Free's namespaces / hooks. New Free must keep every symbol Pro consumes; deleted symbols must have transitional shims or be physically absent from Pro's call graph. |
| 3 | **Pro user, Pro updates first** (Pro auto-update, Free still old) | Old Free is still on disk. New Pro must not require a Free symbol that only exists in the new Free. (Less likely in practice — Pro waits for the Free `Requires Plugins` minimum — but still possible across versions that bump that minimum.) |
| 4 | **Pro user, only Free auto-update** (Pro never updated) | Same as #2; the Pro plugin can stay on its current major for months. |
| 5 | **Pro user, only Pro auto-update** | Same as #3. |
| 6 | **Free user → installs Pro fresh** | Free's runtime must recognise Pro as soon as Pro activates; no stale "Pro absent" state. |
| 7 | **Pro user → deactivates Pro** | Free continues to work end-to-end without any "Pro absent" fatal. No menus, hooks, or shortcodes referencing classes that only exist in Pro. |

### Concrete rules every commit must obey

1. **Never delete a public PHP symbol that Pro currently calls.** If a method must die, leave a thin shim that returns a safe value or `WP_Error` and mark it `@deprecated`. Audit Pro before deletion via:
   `grep -rn "MethodName\|ConstName" ../ar-vr-3d-model-try-on-pro/`.
2. **Never delete a constant Pro references.** Same audit. Constants are even harder to shim — leave them in place and stop *reading* them in Free.
3. **Never change an option-key or post-meta-key name without a migration.** Free owns these keys; Pro reads them. If a rename is unavoidable, ship the rename in Free first with a backward-read fallback that lasts at least one full release before Pro removes the old name.
4. **Never repurpose a hook.** Filters / actions Free exposes (`atlas_ar_*`, `atlasar_*`, `ATLAS_AR_*`) keep their argument shape forever. Add new hooks; don't change existing ones.
5. **`is_pro_active()` must degrade gracefully**, not flip "Pro absent" suddenly. The Free implementation checks the canonical Pro constant first, then falls back to file existence so a half-loaded / upgrading / pre-3.0.0 Pro install still registers as present (see `AR_TRY_ON_Helper::is_pro_active()` code comment for the staged check).
6. **Freemius removal from Free is one-way safe** because Pro's bootstrap re-defines `av3mto_fs()` under a `function_exists` guard and Pro keeps its own `vendor/freemius/` copy. Don't remove that guard or that vendor folder.
7. **REST routes stay registered** even when their old behaviour goes away. Removing a route URL would 404 third-party callers; instead, return a permissive payload (see `can_compress()` in `api/AR_TRY_ON_Compression_Routes.php` — it stayed registered but reports "no cap" now).
8. **React data shape is forward-stable.** `wp_localize_script` payload keys keep their existing types; we may add new keys, never remove or change existing ones, until at least one major release after Pro stops reading them.
9. **Database tables: only Free creates Free's, only Pro creates Pro's.** Don't `CREATE TABLE IF NOT EXISTS` against a Pro-owned table from Free, or vice versa. The pair owns `wp_ar_compression_log` (Free) and `wp_ar_compression_queue` (Pro); these don't move.
10. **Verify in artest before push.** Every commit that crosses the free/pro boundary gets exercised with:
    - Free alone (Pro deactivated)
    - Free + current Pro (both at HEAD)
    - Free at HEAD + Pro at the previous release tag (forward-compat check for scenarios 2 + 4)

### Phase 1 audit result (recorded 2026-05-25)

Phase 1 was audited against the rules above and passed with one safety
improvement applied in the same series:

- `AR_TRY_ON_Helper::is_pro_active()` — added a two-stage check
  (constant first, plugin file fallback) so old/half-loaded Pro builds
  still register as present.
- `av3mto_fs()` removal from Free is safe: Pro's
  `ar-vr-3d-model-try-on-premium.php` declares `av3mto_fs()` under a
  `! function_exists()` guard and bundles its own `vendor/freemius/`
  copy. Pro will define and use its own SDK regardless of what Free
  ships.
- Removed PHP constants (`FREE_USER_LIMIT`, `FREE_FACE_PRODUCT_LIMIT`)
  and methods (`compress_server_side`, `convert_format`,
  `add_to_queue`, `process_queue`, `free_face_product_limit`,
  `count_face_products`, `would_exceed_free_cap`,
  `enforce_free_cap_after_save`, `render_cap_notice`) verified to
  have **no consumers** in the Pro repo (grep across
  `../ar-vr-3d-model-try-on-pro/`).
- Removed filter `atlas_ar_tryon_free_product_limit` — Pro's face
  addon registers an `add_filter` listener for it, but
  `apply_filters('atlas_ar_tryon_free_product_limit', ...)` no longer
  fires anywhere in Free. The Pro `add_filter` call is now a no-op,
  not a fatal. ✓
- `can_user_compress()` kept (not deleted) — returns a permissive
  payload so the React UI's existing fetch keeps working.
- `is_pro_active()` kept on `AR_TRY_ON_Tryon` as a thin Helper-
  delegating wrapper because Pro's face addon and other call sites
  use the local-class name.

## B. The 12-rule checklist mapped to AR-specific code

Each Yoast-pattern rule with concrete actions for this codebase.

| # | Rule | What to do in this repo |
|---|---|---|
| 1 | Free is fully functional standalone | Remove every cap that prevents a Free user from using a shipped feature (sections D + E below). |
| 2 | Pro feature code lives ONLY in Pro | Delete `compress_server_side()`, `convert_format()`, `add_to_queue()`, `process_queue()` from Free's `AR_TRY_ON_Compression`. Delete `wp_ar_compression_queue` migration from Free's activator (Pro creates it). |

| 8 | License logic lives entirely in Pro | Remove the `av3mto_fs()` Freemius call from Free. Pro keeps its own Freemius. |
| 9 | License checks may gate UPDATES, never FEATURES | Free has no license check. Pro's Freemius handles license for updates. |
| 10 | Shared settings framework with clear ownership | Free owns: `ar_try_on_settings`, `ar_try_on_compression_settings`, `ar_try_on_model_data`. Pro adds its own `ar_try_on_pro_*` keys; Pro reads Free's but never writes. |
| 11 | No phoning home from Free without opt-in | Already done — see AR-61 §2 (tracker opt-in, manifest with on-page disclosure, gist removed). |
| 12 | Bundle and disclose assets | Already done — see AR-61 §1.2 (Source code section) and §3.3 (decoders bundled). |

### Hard rule: no file deletions in `src/`

Project owner's instruction: **do not delete any file under `src/`**. Every React file listed in section C as "delete entire file" stays on disk; only the **render output changes** (locked controls → `<PremiumBadge>`). This keeps the build output stable, avoids breaking imports elsewhere, and makes the diff much cleaner.

The same goes for any PHP file Pro might want to extend — we modify in place, never delete.

### Compression behaviour for large files

Free keeps client-side GLB/GLTF compression for ALL file sizes up to **10 MB**. The 5-model count cap is removed entirely — unlimited compressions.

For files **above 10 MB**: Free **refuses to compress** with the user-facing message:

> *"File size is too big — can't be compressed."*

The message does **not** mention Pro. It's framed as a technical limit (browser performance — compressing very large files client-side would lock up the page), not a Free-tier gate. The Pro feature ("server-side compression for large files") is advertised only in the standalone Pro features sidebar, never inline at the point of failure.

Why this matters for wp.org compliance: this is a legitimate technical constraint (real performance reason), not a trialware lock. The Yoast pattern explicitly allows refusing operations when there's a real technical reason — just don't dangle "Pro can do this!" at the user.
